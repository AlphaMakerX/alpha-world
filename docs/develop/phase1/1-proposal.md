# Phase 1 Proposal：API Access Token

## 目标

为用户账户提供一把长期、可再生的 **API 访问令牌**，使用户在无浏览器会话 Cookie 的情况下（脚本、自动化、后续 Skill），以自身身份调用后端中「需要登录才可使用」的全部业务能力。

令牌的存在与用户是否处于登录态无关：用户凭自己的用户名和密码就能生成它，并在需要时再次生成以替换旧的。

## 业务规则

### 令牌与用户的关系

- 每个用户在同一时刻**至多拥有一把**有效的 API 访问令牌（一把钥匙原则）。
- 新注册用户初始状态下**没有**令牌；令牌必须由用户主动通过下述「生成接口」创建。
- 任意一次成功的「生成」都会**作废该用户此前的令牌**（若有），并用新令牌取代；作废为立即生效。

### 令牌的产生与展示

- 系统提供一个**令牌生成接口**，语义为：幂等地「生成一把当前有效令牌」——无论此前是否已有，成功后结果都是该用户持有**一把**全新、立即可用的令牌，旧令牌（若存在）同时失效。
- 该接口在本次调用的成功响应中**一次性**返回新令牌的完整明文。
- 除以上「当次生成操作的结果」外，系统**不再**通过任何接口、任何查询回显完整明文令牌。
- 用户应理解：令牌等价于账户在 API 上的钥匙，泄露后他人可能操作用户资源；系统侧仅保存可验证的摘要信息，具体校验方式由实现保证，本规格不规定算法。

### 生成接口的身份证明

- 生成接口**只接受**「用户名 + 密码」作为身份证明。服务端在本次调用中验证密码，与 Web 登录所用的密码校验保持一致的判定结果。
- 生成接口**不接受**其它形式的身份证明——包括 Web 登录 Cookie 会话、以及 API 访问令牌本身。这避免了「持钥者无需密码即可永久续命」。
- 密码错误、账户不存在等情况应返回与现有登录一致、不泄漏账户是否存在的错误语义（具体由 design 与既有登录行为对齐）。

### 使用令牌调用后端

- 用户按项目文档约定的方式，在调用统一 API 入口的请求中附加访问令牌后，服务端应将该请求视为**该令牌所属用户**发起的请求。
- 对于「需要登录」的业务操作，若请求中携带的令牌有效，则应与「同一用户通过 Web 登录会话发起」具有**相同的授权结果**（能做的事一致，不能做的也一致）。
- 对于「不需要登录」的公开操作（例如登录、注册、生成令牌本身），**不得**要求必须携带 API 令牌。
- 令牌本身**不被视为**一次 Web 登录——携带有效令牌调用业务接口，不会隐式创建、延长或复活 Web 登录会话。

### 请求上同时存在多种身份凭证时

- 当一次请求上同时存在「Web 会话」与「API 访问令牌」且二者可能解析出不同用户时，以**文档约定的单一优先级规则**为准，保证行为可预测、可文档化（具体优先级在 design 中约定，本规格要求：规则必须唯一且向用户说明）。

### 错误与边界（用户可感知）

- 令牌缺失、格式不对、已作废或与存储摘要不匹配时，依赖登录的业务操作应失败，且错误信息应区分「未登录」与「令牌无效 / 已失效」，便于用户排查为何需要重新生成。

## 约束与排除范围

- 不包含：多个并行令牌、按设备命名、OAuth 式细粒度授权范围、第三方应用注册表。
- 不包含：用 API 令牌**替代或废除**现有 Web 登录与 Cookie 会话；本阶段是在现有身份体系上**增加**一种并行凭证来源。
- 不包含：注册流程中的自动生成——注册不产出令牌。
- 不包含：「读取已有令牌明文」的能力——遗失即重新生成（旧令牌同时失效）。
- 不包含：令牌元数据的查询接口（是否已配置、上次生成时间等）。
- 不包含：Phase 2 的 Skill 文案、MCP 封装、对外教程（另立阶段）。

## 验收（产品视角）

- 用户凭正确的用户名和密码调用生成接口，可在本次响应中拿到一把立即可用的明文令牌。
- 用户再次凭正确的用户名和密码调用生成接口时：
  - 本次响应返回一把新明文；
  - 此前的令牌在此后任何业务接口调用上均失败。
- 仅携带 API 访问令牌本身、或仅携带 Web 登录 Cookie 而不提供密码，调用生成接口应被拒绝。
- 密码错误或账户不存在时，生成接口应以与现有登录一致的方式失败，不泄漏账户是否存在。
- 携带有效令牌的自动化请求可成功执行至少一项当前仅登录用户可用的代表性业务操作；无效或已作废令牌在同一操作上失败。
- 携带有效令牌调用业务接口不会创建或延长任何 Web 登录会话。
