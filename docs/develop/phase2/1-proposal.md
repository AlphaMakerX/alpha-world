# Phase 2 Proposal：对外 Agent Skill（API 玩法手册）

## 目标

为「持有本游戏账号 username + password」的**第三方 AI Agent**，产出一份**自包含的 Skill 文档**：Agent 仅凭这一份文档，即可在不读项目源码、不调用浏览器、不复制粘贴 Cookie 的前提下，完成「拿令牌 → 看自己 → 看世界 → 在世界里玩起来（购地、建造、生产、交易）」的端到端闭环。

本阶段**不改服务端代码**：所有依赖能力（API 访问令牌、Bearer 调 tRPC、错误码区分）已在 [Phase 1](../phase1/README.md) 上线。本阶段唯一交付物是一份给 Agent 读的玩法手册。

## 业务规则

### Skill 的形态与读者

- 交付物为**仓库内的一份 Markdown 文档**（位置由 design 决定，例如 `docs/develop/phase2/skill/SKILL.md`），不打包成 SDK、不包装成 MCP、不上线为独立服务。
- 文档的**目标读者是另一个 AI Agent**（LLM 驱动的脚本或助手），不是普通浏览器用户：用语、示例、错误处理须按「机器可解析、可重试」的标准来组织，而非按人类教程的叙事方式来组织。
- 文档**自包含**：Agent 把它整篇读完后，无需再查阅本仓库其它文件即可开始调用接口；所有跳转链接仅作「想深入时的扩展阅读」，不作「执行的必要前置」。

### 前置条件（不在本 skill 范围内）

- Agent **已经持有**一个本游戏账号的 `username` 与 `password`。账号注册（`auth.register`）**不**在 skill 覆盖范围内——如何获取账号由调用方自行解决。
- Agent **能够发起 HTTPS / HTTP POST 请求**并能处理 JSON 响应；不假设它能跑浏览器、能解 Cookie、能执行 JavaScript。
- Agent 运行环境**有持久化能力**（文件、KV、内存任意一种），可用于保存令牌明文到下次调用之间。

### 能力覆盖范围

skill 必须把以下全部 tRPC router 的能力都讲清楚（每个能力包含：用途、是否需要登录、入参约束、典型成功响应形状、典型失败语义）：

- `apiAccessToken.generate`：凭 username + password 生成/覆盖 API 访问令牌（Phase 1 入口）。
- `person.*`：当前账号详情（me）、自己的位置更新、公开的财富排行榜、Adam 与人格 profile。
- `plot.*`：世界地块列表、购买地块。
- `building.*`：在自己的地块上建造、查询自己拥有的全部建筑。
- `inventory.*`：查询自己的物品库存。
- `factory.*`：查看可用配方、查询某工厂的生产订单、启动一笔生产。
- `shop.*`：上架、列出某商店的商品、购买、撤单、查交易历史。
- `purchasingStation.*`：创建求购单、列出某求购站的求购单、卖家履约、撤单、查交易历史。
- `item.*`：物品定义（itemKey / 显示名 / 分类等元数据）。

skill 中对每个能力的描述风格须**对齐**，便于 Agent 用同一套解析逻辑处理；具体格式与字段口径在 design 中固化。

### 调用约定（必须写入 skill）

skill 必须把以下 4 条 tRPC HTTP 通用约定**显式**讲清楚，避免 Agent 因不熟悉 tRPC + superjson 而踩坑：

1. **HTTP 入口**：`POST /api/trpc/<router>.<procedure>`，host 由部署环境决定，本地开发为 `http://localhost:3000`。
2. **请求体形状**：所有入参用 `superjson` 包裹为 `{ "json": <input>, "meta"?: <...> }`；无入参时也要传 `{ "json": null }`。
3. **响应体形状**：成功为 `{ "result": { "data": { "json": <output> } } }`；失败为 `{ "error": { "json": { "message": ..., "code": ..., "data": { ... } } } }`。
4. **认证头**：`Authorization: Bearer <token>`，scheme 大小写不敏感；公开接口可省略此头。

### 凭证管理建议（必须写入 skill）

- 令牌明文**只在生成接口的本次响应里返回一次**，Agent 必须立即落到本地持久化存储。
- 令牌泄漏 / 遗失的处理方式**只有一种**：再调一次 `apiAccessToken.generate`，旧令牌自动失效，新令牌立即可用。
- 同一账号每次生成都会**覆盖**前一把令牌——Agent 在多进程并发场景下需自己负责令牌的串行管理（具体做法不强制）。
- skill 必须明确写出：调用业务接口时**只**需要带 Bearer 头，**不要**同时再带 Cookie 或其它身份信息。

### 错误处理与重试约定（必须写入 skill）

- skill 必须列出 Agent 最可能遇到的错误码与文案，至少包含：
  - `用户名或密码错误`（生成令牌时）→ 检查凭证，**不要**重试。
  - `请先登录` → Agent 忘记带 Bearer，**带上**令牌后重试。
  - `API 令牌无效或已失效` → 立刻调用 `apiAccessToken.generate` 拿新令牌后重试，**不要**反复用旧令牌重试。
  - 业务校验失败（金币不足、库存不足、配方不存在、地块已被占等）→ 由 Agent 在业务层决策，**不**做盲目重试。
- 对网络层错误（5xx、超时）skill 须给出明确建议（指数退避 + 上限），不让 Agent 误把它和「令牌失效」混淆。

### 最小可玩闭环（验收基准）

skill 必须在末尾给出一段**可被 Agent 当作 smoke test 直接执行**的最小闭环示例，覆盖至少：

1. 用 username + password 生成令牌；
2. 拉自己的账号详情（确认令牌生效、知道自己有多少金币、在哪个位置）；
3. 拉世界地块列表（看一眼有什么）；
4. 至少触发一次**写操作**（购地 / 建造 / 上架 / 求购 / 启动一次生产，任选其一），并在响应里看到自身状态变化。

闭环示例的命令风格、行数、是否提供多语言版本（仅 cURL，或 cURL + Node `fetch` + Python）由 design 决定。

## 约束与排除范围

- **不修改服务端代码**：Phase 1 已经把所有依赖能力上线；本阶段仅写文档。如果在写 skill 过程中发现服务端有「不写下来 Agent 就不可能用对」的行为缺陷（如错误码不一致、字段命名歧义），应**回写到 inbox 或 bug 列表**，不在本阶段直接改。
- **不交付 SDK / NPM 包 / Python 包**：保持「一份 Markdown，纯 HTTP 调用」。
- **不交付 MCP server**：是否做 MCP 封装属于另一阶段。
- **不覆盖账号注册**：`auth.register` 不写进 skill，假设 Agent 已有凭证。
- **不覆盖未上线接口**：仅以当前 `appRouter` 实际挂载的 router 为准；新功能上线前不预留章节。
- **不覆盖前端逻辑**：Phaser 场景、UI 组件、客户端缓存等与 Agent 调 API 无关，全部排除。
- **不写翻译版本**：与项目其它文档一致，仅中文。

## 验收（产品视角）

- **一份文档**：`docs/develop/phase2/skill/SKILL.md`（或 design 决定的等价位置）单文件存在，结构清晰，可被一次性读完。
- **能力齐全**：上文「能力覆盖范围」列出的全部 router 在 skill 中均有对应章节，每个章节包含「用途 / 是否需登录 / 入参 / 成功响应 / 失败语义」五项。
- **新 Agent 可独立上手**：把 skill 喂给一个**未读过本仓库源码**的 LLM Agent + 一个有效的 username/password，Agent 应能在无人工介入的前提下完成上文「最小可玩闭环」的全部 4 步。
- **错误恢复正确**：当人为把 Agent 持有的令牌作废（再次调用生成接口），Agent 在下一次业务请求收到 `API 令牌无效或已失效` 时，能按 skill 的指引**自动重新生成令牌并重试**成功，不向用户提问、不无限重试。
- **不诱导越权**：skill 不告诉 Agent 任何「绕过密码续命令牌」「用 Bearer 调 generate」之类的反模式；这些反模式如果在文档里出现，必须以「禁止」的形式出现并解释为什么禁止。
- **与 Phase 1 一致**：skill 中的所有 cURL / 字段名 / 错误文案与 [phase1/README.md](../phase1/README.md) 的「如何携带 Bearer 调用 tRPC」一节**完全一致**，不出现两处事实矛盾。
