# Alpha World

> 给自己和 AI 生成代码用的开发文档。  
> 目标不是写完整策划案，而是提供一份可直接拆任务、直接落地实现的 MVP 规格。

## 1. 文档用途

- 本文档优先服务于 AI 开发执行。
- AI 应优先按照本文档的默认规则实现，不要自行扩展复杂系统。
- 如果实现时发现规则缺失，应先选择最简单、最稳定、最容易落地的方案。

## 2. MVP 目标

本项目是一个基于 `Next.js + React` 的轻量开放世界经营游戏 MVP。

当前阶段的唯一目标是：

- 先做一个可以在浏览器中实际游玩的单机式 Demo
- 所有逻辑都放在同一个 `Next.js` 项目中
- 使用本地 JSON 持久化
- 不追求真实多人联机
- 优先打通完整玩法闭环，而不是追求复杂架构

## 4. 核心产品定义

### 4.1 游戏模式

- 这是一个长期经营型游戏，但 MVP 阶段按单机 Demo 实现
- 玩家登录后进入同一个本地主世界
- 玩家通过掷骰在固定路径上前进
- 玩家通过买地、建造、升级、生产、售卖来积累金币

### 4.2 地图模式

- 不使用二维 `20 x 20` 地图
- 本期地图是一个固定顺序的单环路径
- 地图总格数固定为 `20`
- 玩家沿路径按顺序循环移动
- 走到末尾后回到起点继续前进

### 4.3 账号模式

- 需要实现最简单的注册与登录
- 注册字段最少为：`nickname`、`password`
- 密码使用本地安全哈希存储，不能明文保存
- 所有账号数据保存在本地 JSON
- 登录后支持切换到其他本地账号

## 5. 核心规则

### 5.1 玩家基础数值

- 初始金币：`2000`
- 体力上限：`100`
- 掷骰消耗体力：`10`
- 体力恢复：每 `5` 分钟恢复 `5`
- 在线和离线都恢复体力
- 体力不能超过上限

### 5.2 掷骰与移动

- 骰子点数范围：`1 - 6`
- 玩家点击按钮后执行一次掷骰
- 每次掷骰消耗 `10` 体力
- 如果体力不足，则不能掷骰
- 玩家根据骰子点数沿固定路径前进
- 前进完成后停在落点格子上

### 5.3 落点交互规则

- 落到无主地块：玩家可以手动点击购买
- 落到自己的地块：玩家可以手动查看并决定是否建造或升级
- 落到他人的住宅：自动扣除租金并支付给所有者
- 落到他人的工厂或商店：不自动扣费，但允许查看该地块信息
- 本期没有事件地

### 5.4 地块规则

- 所有可购买地块统一处理，不区分 `zoneType`
- 无主地块可购买
- 已有主人的地块不可直接购买
- 买下地块后，玩家可自由选择建造类型

### 5.5 建筑类型

- `residential`：住宅，用于收租
- `factory`：工厂，用于生产商品
- `shop`：商店，用于出售商品

### 5.6 建造与升级

- 建造是手动触发
- 升级是手动触发
- 建造和升级都立即完成，不需要等待时间
- 只有地块所有者可以建造和升级
- 初始最高等级：`3`

### 5.7 建造成本

- 住宅建造成本：`300`
- 工厂建造成本：`500`
- 商店建造成本：`450`
- 升级成本公式：`upgradeCost(level) = baseBuildCost * level * 0.8`

### 5.8 租金规则

- 只有住宅会收取租金
- 玩家移动到他人住宅时自动结算租金
- 租金公式：`rent = tileBaseRent * buildingLevel`
- 默认 `tileBaseRent = 40`

### 5.9 商品系统

本期只做 2 种商品：

- `wood`
- `brick`

### 5.10 工厂规则

- 工厂由玩家手动点击开始生产
- 每次生产时长：`120` 秒
- 每次生产产出：`5`
- 生产不消耗原料
- 生产完成后增加对应库存

### 5.11 商店规则

- 商店用于把商品卖给系统
- 不做玩家之间的挂单交易
- 售卖为手动触发
- 系统固定价格：
  - `wood = 30`
  - `brick = 45`
- 交易手续费：`5%`

### 5.12 维护与税费

- 地税：每 `30` 分钟收取 `总资产估值 * 1%`
- 建筑维护费：每 `10` 分钟每栋收取 `10 * level`

说明：

- 如果实现复杂度过高，本期可以先保留数据结构和计算函数
- 但至少要保证后续容易补上

## 7. 技术实现约束

### 7.1 项目结构原则

- 所有代码都放在同一个 `Next.js` 项目中
- 前后端逻辑可以通过 `app` 路由、`route handlers` 或服务层实现
- 不拆独立游戏服务器

### 7.2 状态管理原则

- 本期优先使用单进程内存状态 + JSON 持久化
- 所有关键操作都必须先校验再写入
- 不考虑多实例一致性
- 不考虑分布式锁

### 7.3 持久化原则

- 使用本地 JSON 文件保存世界和玩家数据
- 关键动作后立即落盘
- 写入时使用临时文件替换，避免坏档
- 服务启动时如果文件不存在，则自动生成默认数据

### 7.4 安全原则

- 密码必须哈希存储
- 不需要复杂权限系统
- 只要保证基础账号隔离和最小安全性即可

## 8. 数据文件约定

建议使用以下文件：

- `data/world.json`：世界状态、地图配置、系统参数
- `data/players.json`：账号信息、玩家状态、体力、位置、金币、库存
- `data/tiles.json`：地块所有权、价格、建筑状态
- `data/transactions.json`：交易与事件日志

本期可以不单独拆 `market.json`，因为市场规则很简单。

## 13. MVP 验收标准

- 能注册新账号并登录
- 能查看并切换本地账号
- 能进入主游戏页并看到 20 格环形地图
- 能掷骰并沿固定路径移动
- 能在无主地块上手动购买
- 能在自己的地块上手动建造和升级
- 能在进入他人住宅时自动扣租
- 能启动工厂生产并在完成后获得商品
- 能通过商店把商品卖给系统并获得金币
- 所有关键数据在重启后可以恢复
- 不出现负金币、重复买地、重复扣费、重复领取产物等严重错误

## 14. AI 实现原则

- 不要提前实现真实多人架构
- 不要引入数据库
- 不要引入复杂消息队列
- 不要为了未来扩展而把 MVP 做得过重
- 优先保证规则闭环、代码清晰、状态可恢复

如果文档没有写到某个细节，默认选择最简单方案，并保持实现易于后续迭代。


P0（先做，直接提升体验）
[ ] 骰子点数展示效果
需求：掷骰后清晰展示“掷出了几”（数字 + 动画/高亮）
建议：加一个 0.8~1.2s 的骰子滚动动画，结束后定格点数
验收：每次掷骰都能明确看到最终点数，且不会与后续移动冲突
[ ] 人物逐格移动效果
需求：角色按点数一格一格移动，而不是瞬移
建议：每格 200~350ms，支持中途经过格子触发轻反馈（高亮/音效）
验收：掷出 N 点时，角色连续移动 N 次，最终位置与逻辑一致



P1（核心玩法补充）
[ ] 增加生产材料系统
需求：新增材料类型（如木材、石料、铁矿、粮食）及产出逻辑
建议：先做基础闭环：地块/建筑 -> 定时或回合产出 -> 入库
验收：至少 3 种材料可稳定产出、存储、在 UI 中可见
P2（内容扩展）
[ ] 丰富地图内容
需求：扩充地块类型与地图事件，提升可玩性
建议：
新增地形：资源地、事件地、交易地、风险地
每类地块至少 1 个可触发效果
验收：地图可探索内容明显增加，单局体验不重复
